1/*
2 Implementation notes:
3
4 - An OOP style is used where each "class" is made up of a struct and a number of functions prefixed with the struct name.
5
6 - struct fields that are const are readonly. Either they are set in a create function and can never be changed, or they can only
7 be changed by calling a function.
8
9 - Inheritance is done using a struct field named "super" as the first field, allowing the struct to be cast to its "super class".
10 This works because a pointer to a struct is guaranteed to be a pointer to the first struct field.
11
12 - Classes intended for inheritance provide init/deinit functions which subclasses must call in their create/dispose functions.
13
14 - Polymorphism is done by a base class providing function pointers in its init function. The public API delegates to this
15 function.
16
17 - Subclasses do not provide a dispose function, instead the base class' dispose function should be used, which will delegate to
18 a dispose function pointer.
19
20 - Classes not designed for inheritance cannot be extended because they may use an internal subclass to hide private data and don't
21 expose function pointers.
22
23 - The public API hides implementation details, such as init/deinit functions. An internal API is exposed by extension.h to allow
24 classes to be extended. Internal functions begin with underscore (_).
25
26 - OOP in C tends to lose type safety. Macros for casting are provided in extension.h to give context for why a cast is being done.
27
28 - If SPINE_SHORT_NAMES is defined, the "sp" prefix for all class names is optional.
29 */
